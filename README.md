# engagement-risk-ml

**End-to-end customer churn prediction pipeline for online retail — from raw transaction logs to a deployed weekly scoring script — achieving PR-AUC of 0.823 with XGBoost.**

## Project Overview

Customer churn is one of the highest-leverage problems in retail analytics. Identifying at-risk customers before they disengage — and acting on that signal with targeted re-engagement — is consistently more cost-effective than acquiring new customers. This project builds a complete, production-oriented churn prediction system on the [UCI Online Retail II dataset](https://archive.ics.uci.edu/dataset/502/online+retail+ii), a real-world transactional dataset covering ~525,000 orders across ~4,300 customers.

The project is structured as a professional ML pipeline rather than a standalone analysis. It covers the full lifecycle: data cleaning, leakage-safe feature engineering using time-series snapshots, baseline heuristic evaluation, logistic regression modeling, threshold tuning, regularization analysis, tree-based model comparison, and a production-ready weekly scoring script. Every modeling decision is grounded in business context — the goal is not just to build a model that scores well, but to build one that can be deployed and trusted.

**Key result:** XGBoost with tuned hyperparameters achieves ROC-AUC of 0.715 and PR-AUC of 0.823, operating at a classification threshold of 0.33 — catching 90.3% of at-risk customers at 75.8% precision, recommended for low-cost re-engagement channels such as email and SMS.

## Methodology

### 1. Data Cleaning and Order Table Construction
Raw transaction data is cleaned by removing records with missing customer IDs, filtering to positive purchases only (excluding returns and cancellations), and computing per-line revenue. Transactions are aggregated to an order-level table capturing order timestamp, total revenue, quantity, and item diversity per invoice.

### 2. Snapshot-Based Labeling — Preventing Temporal Leakage
A critical design decision in any time-series ML project is preventing **data leakage** — the accidental use of future information to predict the past. This project uses a **snapshot-based labeling approach**: for each weekly snapshot date, features are computed exclusively from orders at or before that date, and the churn label is defined by whether the customer makes a purchase in the 30 days following the snapshot. Every row in the modeling dataset represents "what did we know about customer C at time T, and did they churn in the next 30 days?" The train/test split further enforces temporal integrity by using the final 20% of snapshots as the held-out test set rather than random sampling.

### 3. Feature Engineering
Twelve behavioral features are constructed across three time windows:

| Window | Features |
|---|---|
| 7-day (short-term) | orders_7d, revenue_7d, items_7d |
| 30-day (medium-term) | orders_30d, revenue_30d, items_30d |
| 90-day (long-term) | orders_90d, revenue_90d, items_90d |
| Lifetime | lifetime_orders, lifetime_revenue |
| Recency | recency_days |

The churn base rate in the dataset is approximately 69% — a highly imbalanced problem that motivates careful metric selection and class-weight balancing throughout the modeling work.

### 4. Baseline Evaluation
Two baselines are established before any ML modeling: a recency heuristic (predict churn if no purchase in last 30 days) and an all-churn classifier. The recency heuristic achieves only 52% accuracy — worse than the all-churn baseline — motivating a learned model that incorporates frequency and monetary signals alongside recency.

### 5. Logistic Regression Baseline
A logistic regression pipeline with StandardScaler and balanced class weights is trained and evaluated. Multicollinearity diagnostics (Pearson correlation matrix and Variance Inflation Factor analysis) reveal significant collinearity among the overlapping time-window features, explaining counterintuitive positive coefficients on short-window order features. Lift analysis demonstrates the model achieves approximately 84% of the theoretical maximum lift given the high base churn rate.

### 6. Threshold Tuning
The default classification threshold of 0.5 is shown to be suboptimal — the max F1 threshold is 0.21, substantially lower due to balanced class weights shifting the model's probability outputs. Three business operating points are identified and compared: Balanced (Max F1), High Recall for broad outreach, and High Precision for targeted interventions.

### 7. Regularization Analysis
L1 (Lasso), L2 (Ridge), and ElasticNet penalties are compared across six orders of magnitude of regularization strength. All three penalties converge to equivalent performance past C=0.01, confirming the baseline model is stable and well-calibrated. L1 at its optimal C eliminates 4 of 12 features — revenue_7d, orders_30d, revenue_30d, and lifetime_revenue — achieving equivalent performance with a sparser model that directly validates the multicollinearity findings.

### 8. Tree-Based Model Comparison
Random Forest and XGBoost are trained using RandomizedSearchCV with TimeSeriesSplit cross-validation, optimizing for PR-AUC. Random Forest matches but does not beat logistic regression, suggesting the signal is predominantly linear and not well-suited to parallel ensemble averaging. XGBoost outperforms both with a well-regularized shallow-tree configuration, confirming that sequential error-correction extracts residual nonlinear signal that parallel ensembles and linear models both miss.

### 9. Stacking Viability Analysis
Error correlation analysis across all three models rules out stacking as a viable next step. Logistic regression and XGBoost share an error correlation of 0.756, and 16.09% of test set customers are misclassified by all three models simultaneously — a hard core of ambiguous cases that no ensemble can resolve with the current feature set. The path to further improvement lies in richer behavioral features rather than more sophisticated ensembling.

## Results

### Model Comparison

| Model | ROC-AUC | PR-AUC |
|---|---|---|
| Logistic Regression (Baseline) | 0.699 | 0.810 |
| Random Forest (Tuned) | 0.693 | 0.813 |
| **XGBoost (Tuned)** | **0.715** | **0.823** |

### XGBoost Business Scenario Comparison (Threshold Tuning)

| Scenario | Threshold | Precision | Recall | F1 | % Flagged |
|---|---|---|---|---|---|
| Balanced (Max F1) | 0.21 | 72.9% | 97.1% | 0.833 | 91.8% |
| **High Recall — Broad Outreach** | **0.33** | **75.8%** | **90.3%** | **0.824** | **82.2%** |
| High Precision — Targeted Outreach | 0.73 | 85.1% | 31.1% | 0.456 | 25.2% |

**Recommended deployment configuration:** XGBoost at t=0.33 for e-commerce re-engagement campaigns using low-cost outreach channels.

## Key Findings

- **The default 0.5 threshold is substantially suboptimal.** Training with balanced class weights shifts the model's probability outputs such that the max F1 threshold is 0.21 for both logistic regression and XGBoost. Threshold tuning is always necessary — never assume 0.5.

- **L1 regularization confirmed the multicollinearity diagnosis.** VIF analysis identified severe collinearity among the overlapping time-window features (orders_90d VIF=15.5). L1 independently eliminated the four most redundant features, achieving equivalent performance with 8 of 12 features — directly validating the diagnostic findings.

- **Random Forest's feature importance is unreliable under multicollinearity.** lifetime_revenue was ranked the single most important feature by Random Forest (0.297 Gini importance) despite being zeroed out by L1. When correlated features compete for splits, Gini importance distributes arbitrarily between them — a well-documented limitation that causal interpretation requires caution around.

- **The right kind of complexity matters more than the amount.** Random Forest with unlimited tree depth failed to beat logistic regression. XGBoost with shallow trees and a low learning rate succeeded. Sequential error-correction extracts nonlinear signal more efficiently than parallel averaging when the signal is predominantly linear.

- **Stacking is ruled out by error correlation, not assumption.** A quantitative error correlation analysis shows LR/XGBoost errors correlate at 0.756, and 16.09% of customers are misclassified by all three models — confirming that the performance ceiling is a feature limitation, not a modeling limitation.

- **The model achieves ~84% of theoretical maximum lift** given the ~69% base churn rate. With a high base rate, the headroom for lift is mathematically constrained — framing performance relative to the theoretical ceiling is a more honest and informative evaluation than raw lift magnitude.

## Why No Neural Network?

Neural networks were deliberately excluded from this project's model 
comparison. On structured tabular data with a small number of engineered 
features, gradient boosting consistently outperforms neural networks — 
this is one of the most well-established empirical findings in applied 
ML, supported by extensive benchmarking on real-world tabular datasets. 
Neural networks derive their power from learning representations from 
raw, high-dimensional inputs (pixels, text, audio) where feature 
engineering is impractical. With 12 hand-engineered behavioral features 
and ~80,000 rows, there is no representational complexity for a neural 
network to exploit that XGBoost cannot already capture more efficiently 
through gradient boosting.

## Setup and Usage

### Requirements

```bash
git clone https://github.com/eugenefkim/engagement-risk-ml
cd engagement-risk-ml
pip install -r requirements.txt
```

### Data

Download the [UCI Online Retail II dataset](https://archive.ics.uci.edu/dataset/502/online+retail+ii) and place the `.xlsx` file in `data/raw/`.

### Running the Notebooks

Execute notebooks in order — each notebook persists intermediate outputs to `data/processed/` for the next to load:

```
01_eda.ipynb
02_labeling_and_features.ipynb
03_modeling_logistic_regression.ipynb
04_threshold_tuning_and_regularization.ipynb
05_tree_based_models_and_model_selection.ipynb
```

### Running the Scoring Script

```bash
python src/score.py
```

The script will prompt you to configure the prediction horizon, feature windows, and recency threshold interactively before scoring all active customers and outputting a ranked CSV of churn probabilities.

## Future Work

The error correlation analysis confirms that the current performance ceiling is driven by feature limitations rather than modeling choices. Meaningful improvement beyond XGBoost's current PR-AUC of 0.823 would require richer behavioral signals — browsing and session-level engagement data, promotional response history, customer support interactions, or product category affinity. Given sufficient feature enrichment, revisiting gradient boosting or a well-regularized neural network on the expanded feature set would be the natural next step. On the deployment side, wrapping `score.py` in a REST API (FastAPI) and containerizing it with Docker would complete the path from research prototype to production microservice.
